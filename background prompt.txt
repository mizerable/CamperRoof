### ADDENDUM: Architectural Decisions & Clarifications (Added post-review)

1. **Upper Limit Calibration**: When hitting UP in the SET state, we ONLY update the `UPPER_LIMIT` to the max position. We do NOT overwrite the `CURRENT_POSITION` variables.
2. **Lower Limit Calibration & Manual Jogging**: Because the Cytrons allow manual jogging via onboard buttons, the roof can be manually leveled at the bottom. When hitting DOWN in the SET state, setting all `CURRENT_POSITION` variables to 0 perfectly syncs the logical state to the newly established physical baseline. (The PCNT hardware continues to perfectly track pulses even during manual jogging).
3. **Button Priority**: State transitions strictly follow the defined button combinations. Any unspecified combination (e.g., pressing UP and DOWN simultaneously) is implicitly ignored and does not cause a state transition.
4. **PCNT Rollover Handling**: The ESP32 PCNT registers are 16-bit signed integers. We elegantly handle hardware rollovers (without missing pulses or using interrupts) by subtracting the previous 16-bit read from the current 16-bit read and casting to `int16_t`. This standard C math inherently resolves the exact delta distance.
5. **Motor Control Protocol**: The Cytron MDDS30s will be configured for "Simplified Serial Mode" at 9600 baud. The software uses an abstract -100 to 100 throttle internally, and translates it to the 0-255 raw unsigned bytes expected by the Left and Right Cytron channels.
6. **Data Structures**: The `targetPosition` and `isFaulted` variables were discarded. The floating midpoint algorithm is purely reactive, and `STATE_FAULT` inherently covers the fault flag.
7. **Boot Sequence**: The system boots directly into the `WAIT` state, and immediately reads the stored positions and limits from the I2C FRAM into RAM.

Bottom-Out Protection (STATE_BOTTOMED): When lowering the roof, if any individual motor moves significantly slower than expected (e.g. bogging down against the seal), that specific motor must be instantly halted. If the roof diverges by >500 pulses as a result, it will trigger STATE_FAULT. If all 4 motors bottom out and halt, the system must transition to STATE_BOTTOMED.
15. Bottom-Out Power Resilience: The individual `is_bottomed_out` flags for all 4 motors MUST be written to FRAM. If the system loses power while bottomed out, it must boot up, read those flags, and if the user presses DOWN, it must instantly transition back to STATE_BOTTOMED without ever applying throttle, completely preventing damage to the seals. Bottom-out flags are cleared whenever the user lifts the roof (STATE_LIFTING).

16. Manual Motor Jogging via Controller: To eliminate the need to use physical buttons on the Cytrons, a fifth button called MOTOR_SELECT is added. Pressing it only has an effect in the WAIT and FAULT states. Starting from WAIT, pressing it cycles through states MOTOR1, MOTOR2, MOTOR3, MOTOR4, and back to WAIT. Starting from FAULT, it cycles through MOTOR1, MOTOR2, MOTOR3, MOTOR4, and back to FAULT. When in a MOTORx state, only the UP button and DOWN button have an effect. Pressing UP lifts that specific motor at a very low 5% throttle, and DOWN lowers it at 5% throttle. This movement strictly respects the upper and lower travel limits, unless the CLEAR button is simultaneously held.
